8.1 Conda: Isolated Environments for Reproducible Science

Bioinformatics pipelines depend on many software packages, each with specific version requirements. A RNA-seq pipeline might require Python 3.10, STAR 2.7.11, and FastQC 0.12 — while a network analysis pipeline might require conflicting versions of the same libraries. Without environment management, installing one pipeline's dependencies can silently break another, a problem known as "dependency hell."

Conda solves this by creating completely isolated environments, each with its own Python version, R version, and package set, that cannot interfere with each other.

💡 Why This Matters for Reproducibility

A Conda environment can be exported to a single YAML file and recreated identically on any machine. Published papers increasingly require sharing environment files alongside code so others can reproduce the exact analysis. NASA OSDR consensus pipelines provide environment files for exactly this reason.

8.1.2 Installing Miniconda

Miniconda is a minimal Conda installer for Linux servers. Install as follows:

# Step 1: Download the installer
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh

# Step 2: Run the installer
bash Miniconda3-latest-Linux-x86_64.sh

# Step 3: Activate Conda in the current shell
source ~/.bashrc

# Step 4: Verify installation
conda --version

A version number such as "conda 24.x.x" confirms success. The prompt will now show "(base)".

8.2 Managing Conda Environments

8.2.1 Creating Environments

CommandDescription
conda create -n myenv python=3.10Create an environment named "myenv" with Python 3.10
conda create -n rnaseq python=3.10 numpy pandasCreate with specific packages pre-installed
conda env create -f environment.ymlCreate an environment from a shared YAML file
conda activate myenvActivate an environment (prompt changes to show the environment name)
conda deactivateReturn to the base environment
conda env listList all environments; active one marked with *
⚠️ Always Activate Before Installing

Installing packages without activating the correct environment installs them into base, causing conflicts. Always confirm the environment name is shown in your prompt before running any install command.

8.2.3 Installing Packages

CommandDescription
conda install numpyInstall a package into the active environment
conda install -c bioconda star=2.7.11aInstall from the bioconda channel (for bioinformatics tools)
conda install -c conda-forge -c bioconda fastqcInstall with multiple channels specified
pip install package_nameInstall packages from PyPI (for tools not in conda channels)
conda listList all packages in the active environment
conda env remove -n myenvCompletely delete an environment
💡 Bioconda Channel for Bioinformatics

The bioconda channel hosts thousands of bioinformatics tools: FastQC, STAR, HISAT2, SAMtools, featureCounts, and more. Always add conda-forge alongside bioconda: conda install -c conda-forge -c bioconda <tool>. Search for available tools at anaconda.org/bioconda.

8.2.4 Exporting and Sharing Environments

# Export the active environment to a YAML file
conda env export > environment.yml

# Recreate an environment from the YAML file on any machine with Conda
conda env create -f environment.yml

8.3 Git and GitHub: Version Control for Science

Version control tracks every change made to analysis scripts, with a complete history of who changed what and when. For bioinformatics this enables: experimenting with new analysis parameters without fear of losing working code; rolling back to any previous version; collaborating with lab members without overwriting each other's work; and sharing the exact analysis pipeline alongside a published paper.

💡 Git vs. GitHub

Git is the version control system — it runs locally on any machine and tracks changes to files. GitHub is a cloud platform that hosts Git repositories online, enabling backup, sharing, and collaboration. Git can be used without GitHub (local only), but using both together is the standard scientific workflow.

8.3.2 Installing and Configuring Git

# Install Git
sudo apt update && sudo apt install git
git --version  # verify installation

# Configure identity
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
git config --list  # verify settings

8.3.3 Connecting to GitHub via SSH Keys

# Step 1: Generate an SSH key pair
ssh-keygen -t ed25519 -C "your.email@example.com"

# Step 2: Start the SSH agent and add your key
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Step 3: Copy your public key (then add it to GitHub Settings → SSH Keys)
cat ~/.ssh/id_ed25519.pub

# Step 4: Test the connection
ssh -T git@github.com

The Daily Git Workflow

# Pull latest changes before starting work
git pull

# Make your changes to files, then stage them
git add script.R  # stage a specific file
git add .         # stage all changed files

# Commit with a descriptive message
git commit -m "Add DESeq2 normalization step to pipeline"

# Push to GitHub
git push
