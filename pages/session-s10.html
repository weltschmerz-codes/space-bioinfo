9A.1 Pipeline Overview & Study Design

This two-session walkthrough reproduces the NASA GeneLab GLDS-525 consensus RNA-seq pipeline from raw data to biological insight. The study is RR-23: bulk RNA-sequencing of mouse cerebellum tissue from animals that spent time aboard the International Space Station, with the scientific question — which genes change their expression in response to spaceflight?

💡 Study at a Glance

Organism: Mus musculus (mouse)  ·  Tissue: Cerebellum  ·  Samples: 29 (Space Flight, Ground Control, Vivarium Control)  ·  Genome: Ensembl GRCm39 release 112 + ERCC92 spike-ins  ·  Sequencing: Paired-end, 150 bp

GroupAbbreviationDescription
Space FlightFLTMice housed on the ISS during the mission
Ground ControlGCMatched ground-based habitat controls
Vivarium ControlVIVStandard vivarium housing baseline

9A.3 Downloading FASTQs from NASA GeneLab

Raw data lives on the NASA GeneLab Open Science Data Repository (OSDR). Each of the 29 samples has two FASTQ files (R1 and R2 for paired-end reads), for a total of 58 files. Downloads are parallelised using Python's ThreadPoolExecutor and use wget --continue for resume support.

# Files are named following GeneLab convention:
# GLDS-525_rna-seq_{SampleName}_R1_raw.fastq.gz
# GLDS-525_rna-seq_{SampleName}_R2_raw.fastq.gz

# Download is driven by the runsheet CSV:
RUNSHEET_PATH = DIRS['runsheet'] / 'GLDS-525_rna_seq_bulkRNASeq_v2_runsheet.csv'
💡 What is a FASTQ file?

A FASTQ file contains millions of short DNA sequences ("reads", typically 150 bp) from the sequencer. Each read has four lines: a header, the DNA sequence, a separator, and per-base quality scores encoded as ASCII characters. Paired-end sequencing produces two files per sample (R1 and R2) that cover both ends of each DNA fragment.

9A.4 Raw Read Quality Control with FastQC & MultiQC

FastQC examines each FASTQ file and generates an HTML report covering per-base quality scores, GC content, adapter content, and sequence duplication levels. MultiQC then combines all 58 individual reports into a single interactive dashboard.

FastQC ModuleWhat to Look For
Per-base sequence qualityGreen zone (Q>28) across most of the 150 bp read
Per-sequence GC contentBell curve centered near 50% for mouse
Adapter contentSome signal expected at 3′ end — will be removed by TrimGalore
Sequence duplicationSome duplication normal for RNA-seq; overduplication may indicate PCR artifacts

9A.5 Adapter Trimming with TrimGalore

TrimGalore (wrapping Cutadapt) removes Illumina adapter sequences and low-quality 3′ bases. The pipeline then runs a second FastQC + MultiQC pass to verify that adapter content has dropped to near zero.

# TrimGalore paired-end command (GeneLab parameters):
trim_galore \
  --paired \
  --phred33 \
  --cores 4 \
  --gzip \
  {R1} {R2} \
  -o {trimmed_dir}
💡 Post-trim QC Checklist

Compare the before/after MultiQC reports. You should see: adapter content → near zero; per-base quality remaining high; sequence length distribution now showing a slight spread below 151 bp (trimmed reads).

9A.6 Genome Indexing & Alignment with STAR

Before alignment, we need a reference genome. We download the Ensembl GRCm39 FASTA and GTF annotation, then concatenate the ERCC92 spike-in sequences used by GeneLab to create a combined reference. STAR requires a pre-built index of this combined reference.

9A.6.1 Building the STAR Genome Index

STAR \
  --runMode genomeGenerate \
  --genomeDir {star_index_dir} \
  --genomeFastaFiles {merged_fa} \
  --sjdbGTFfile {merged_gtf} \
  --sjdbOverhang 150 \
  --runThreadN {cores}
# ⏱ Takes 1–2 hours · Requires ~32 GB RAM · Only done once
💡 Why ERCC92 Spike-ins?

ERCC92 are synthetic RNA molecules of known concentrations added to samples before library prep. Including them in the reference allows quality checks on library preparation efficiency and can be used for normalization. GeneLab includes them in all NASA consensus pipelines.

9A.6.2 Alignment Output & BAM Processing

STAR produces two BAM files per sample: a coordinate-sorted BAM for genome browsers and a transcriptome-aligned BAM passed to RSEM. SAMtools sorts and indexes the coordinate BAM for fast random access.

9B.1 RSEM Gene Quantification

Alignment (Session 9A) tells us where each read maps. RSEM answers the harder question: how many reads came from each gene? This is tricky because some reads map equally well to multiple locations (multi-mapping). RSEM uses an Expectation-Maximization (EM) algorithm to probabilistically assign these reads.

💡 RSEM vs. featureCounts

Simple counting tools (featureCounts, HTSeq) discard multi-mapping reads. RSEM instead assigns them fractionally according to their estimated likelihood, recovering more information — especially important for genes in families with paralogues. NASA GeneLab consensus pipelines use RSEM for this reason.

# Step 1: Build RSEM reference index (once, ~30–60 min)
rsem-prepare-reference \
  --gtf {merged_gtf} \
  {merged_fa} \
  {rsem_ref_prefix}

# Step 2: Quantify each sample
rsem-calculate-expression \
  --bam \
  --paired-end \
  --strandedness forward \
  --estimate-rspd \
  --append-names \
  --no-bam-output \
  -p {threads} \
  {toTranscriptome_bam} \
  {rsem_ref_prefix} \
  {output_prefix}

9B.1.2 Assembling Count Matrices

Each sample produces a .genes.results file. The pipeline extracts the expected_count column from each file and assembles them into a single matrix (rows = genes, columns = samples). A separate TPM matrix is built for visualization.

MatrixColumn UsedUsed For
Raw count matrixexpected_countDESeq2 differential expression
TPM matrixTPMVisualization, cross-sample comparison

Before running DESeq2, the pipeline performs two cleaning steps: removing rRNA genes (which can overwhelm normalization) and merging technical replicates by summing counts from the same biological sample sequenced twice.

9B.2 Differential Expression with DESeq2

DESeq2 is the gold-standard method for count-based differential expression. The pipeline uses PyDESeq2, a Python implementation. Three pairwise contrasts are tested: FLT vs GC, FLT vs VIV, and GC vs VIV.

DESeq2 StepWhat It Does
Size factor normalizationCorrects for different sequencing depths between samples
Dispersion estimationModels biological variability per gene; borrows information across genes
Negative binomial GLMTests whether the group effect is statistically significant per gene
Multiple testing correctionBenjamini-Hochberg FDR; uses adjusted p-value (padj) ≤ 0.05
 
# Pre-filter: keep genes with ≥10 total counts across all samples
# (GeneLab criterion — reduces tests, speeds up computation)
filtered_counts = counts_df[counts_df.sum(axis=1) >= 10]

# DESeq2 significance thresholds (matching GeneLab):
LOG2FC_CUTOFF = 1.0     # |log₂ fold change| > 1 (2× change)
PADJ_CUTOFF   = 0.05    # Adjusted p-value < 5%

9B.2.1 Volcano Plots

A volcano plot shows every tested gene as a dot: x-axis = log₂ fold change, y-axis = −log₁₀(adjusted p-value). Significant DEGs appear in the upper corners of the "eruption". Genes passing both thresholds are colored red (upregulated) or blue (downregulated).

💡 Reading a Volcano Plot

The horizontal dashed line marks padj = 0.05. The two vertical dashed lines mark |log₂FC| = 1. Only genes in the upper-left and upper-right quadrants are considered statistically significant with biologically meaningful effect size. Genes near the x-axis are highly significant but show small fold changes — likely artifacts of very large sample sizes.

9B.2.2 Principal Component Analysis (PCA)

PCA collapses thousands of gene expression values into 2–3 dimensions. Samples from the same group should cluster together; well-separated clusters between FLT and GC indicate a strong spaceflight transcriptomic signature. The pipeline uses VST (Variance-Stabilizing Transformation) normalized counts for PCA, which prevents highly expressed genes from dominating the analysis.

9B.2.3 Heatmaps of Top DEGs

Heatmaps display the top 50 DEGs per contrast as a color grid (rows = genes, columns = samples). Z-score normalization is applied per gene so expression patterns are comparable across genes of very different absolute expression levels.

9B.3 Pathway Analysis: ORA & GSEA

A list of DEGs is just the beginning. The real biology emerges when we ask whether those genes cluster into known functional categories.

MethodInputQuestion AskedAdvantage
ORA (Over-Representation Analysis)Significant DEG listAre DEGs enriched in specific pathways vs. what random chance predicts?Simple, intuitive, fast
GSEA (Gene Set Enrichment Analysis)All genes ranked by log₂FCDo pathway genes cluster at the top or bottom of the ranked list?Uses all genes; detects coordinated subtle shifts
💡 Gene Ontology (GO) and KEGG

Both ORA and GSEA are run against two databases: Gene Ontology (GO), which classifies genes by Biological Process, Molecular Function, and Cellular Component; and KEGG, which maps genes onto metabolic and signalling pathways. Results from both are included in the Notebook 4 output.

# GSEA workflow:
# 1. Rank all ~20,000 expressed genes by log₂FC (most up → most down)
# 2. For each gene set, walk the ranked list and compute an enrichment score
# 3. Permute sample labels to estimate statistical significance
# 4. Report Normalized Enrichment Score (NES) and FDR q-value

The final step validates our entire pipeline by correlating our log₂ fold changes against the official GeneLab pre-computed results. A Pearson r > 0.99 confirms the analysis is fully reproducible.
